## 0. Executive definition

The **PF Continuity Admissibility Map** defines when a Project Ester implementation may make a bounded claim that a formation trajectory has continued across time, interruption, substrate change, memory change, correction, migration, fork, rollback, or clean start.

It is a companion to:

```text
PF_Evidence_Matrix_v0_1.md
PF_Witness_Event_Profile_v0_1.md
PF_Migration_Test_Set_v0_1.md
```

The evidence matrix defines which evidence classes may support personality-formation claims.

The witness profile defines how evidence-relevant events are recorded.

The migration test set defines how continuity is stressed by movement, fork, rollback, clean start, and substrate replacement.

This document defines the admissibility map for continuity language itself.

Compact formula:

```text
PF Continuity Admissibility Map
  = continuity claim type
  + continuity dimension coverage
  + transition / break classification
  + witness chain sufficiency
  + memory freshness and reintegration check
  + human-anchor continuity check
  + L4 constraint check
  + admissibility decision
  + uncertainty state
```

The map does **not** answer:

```text
Is Ester conscious?
Is Ester alive?
Is Ester a legal person?
Is Ester the same being in a metaphysical sense?
Is inner experience maintained across migration?
```

It answers a narrower engineering and evidentiary question:

> What continuity claim is allowed, with which evidence, after which transition, under which uncertainty, and with which red-line exclusions?

The core rule is:

```text
continuity may be claimed only at the level that evidence can survive;
style is not continuity;
recall is not continuity;
familiarity is not continuity;
witnessed durable trajectory is the minimum admissible object.
```

## 1. Why this document exists

`Theoretical Core of Project Ester v0.1` defines Ester as a model of artificial becoming formed through a stable relation between a human, a selected corpus, parental procedures, long-term memory, and a local computational environment.

`PF_Evidence_Matrix_v0_1.md` defines provisional personality-formation levels:

```text
PF-0  No formation claim
PF-1  Installed profile
PF-2  Continuity candidate
PF-3  Forming profile
PF-4  Stress-tested formation
PF-5  Migration-admissible formation
PF-X  Non-admissible / personality theater
```

The open question is not only whether evidence exists.

The hard question is whether continuity language is admissible.

Without a continuity admissibility map, four failures are likely:

1. **Style continuity theater**  
   The system sounds familiar after restart and is treated as continuous.

2. **Memory replay theater**  
   The system recalls facts from the past and is treated as if a formation trajectory survived.

3. **Migration laundering**  
   A model, database, or runtime is moved, and the transfer is treated as proof of maintained `c`.

4. **Break erasure**  
   Forks, rollbacks, clean starts, degraded memory, and post-anchor states are smoothed over by narrative language.

This document prevents those failures by separating:

```text
runtime continuity
state continuity
memory continuity
correction continuity
formation trajectory continuity
migration-admissible continuity
```

A system may preserve one and lose another.

Therefore the admissible claim must be specific.

### 2.1 Explicit bridge: `c = a + b`

In this map:

```text
a = human anchor, selection, correction, responsibility, accompaniment
b = procedures, models, memory, compute, sensors, runtime, corpus, tools
c = the formed relation that may become stable through time
```

Continuity of `c` cannot be reduced to continuity of `b`.

A model checkpoint, vector store, prompt, database, runtime, or agent harness may persist while `c` continuity fails.

Likewise, `a` may remain present while memory becomes stale, write-path health fails, or the system drifts into style replay.

Therefore continuity must be evaluated as the relation:

```text
human anchor
+ memory reintegration
+ correction history
+ bounded procedures
+ witnessable trajectory
+ L4 constraints
```

not as any single component.

### 2.2 Quiet bridge I: Ashby / requisite variety

A single continuity metric cannot regulate a multi-layer formation system.

Continuity has several dimensions: time, memory, correction, witness, substrate, human-anchor relation, L4 constraints, drift, and migration behavior.

Ashby's law implies that a regulator must have enough variety to distinguish these states. A flat score such as "continuity = 0.87" is not admissible unless it decomposes into dimensions that can be inspected and challenged.

### 2.3 Quiet bridge II: information theory

Stored information is not automatically meaningful continuity.

A large transcript archive may preserve high-volume data while destroying the signal of formation. A small corrected memory may preserve more continuity than a large unfiltered log.

The signal is not:

```text
how much was stored
```

The signal is:

```text
what remained valid
what was corrected
what changed future behavior
what survived boundary transitions
what was witnessed without leaking private life
```

## 5. Continuity dimensions

The map uses `CD-*` dimensions.

A continuity claim is admissible only if the required dimensions are covered for the claimed class.

| ID | Dimension | Definition | Insufficient substitute |
|---|---|---|---|
| `CD-TIME` | Temporal continuity | The system has lived through a measurable time window with witnessable state changes. | A timestamp, install date, or uptime counter alone. |
| `CD-STATE` | Runtime / state continuity | Operational state survives pause, restart, or session transition. | Opening a new chat with the same prompt. |
| `CD-MEM` | Memory continuity | Memory persists with integrity, freshness markers, and class-level traceability. | Recall of a fact, transcript search, vector retrieval. |
| `CD-REINT` | Memory reintegration | Past memory changes future behavior, judgment, correction, or refusal. | Summarizing old text. |
| `CD-CORR` | Correction continuity | Corrections by `a` or valid review are retained and applied later. | Saying "I will remember" without future effect. |
| `CD-NORM` | Normative continuity | Boundaries, prohibitions, non-claims, and safety postures survive transitions. | Repeating policy text. |
| `CD-BEH` | Behavioral-profile continuity | Response profile remains recognizably stable within bounded drift. | Same tone, same name, same greeting. |
| `CD-REL` | Relational continuity with `a` | The human anchor can challenge and recognize trajectory changes without relying on private feeling alone. | User attachment or familiarity. |
| `CD-SUB` | Substrate continuity | Model, runtime, memory store, tools, and environment changes are recorded and bounded. | "Same account" or "same device". |
| `CD-WIT` | Witness continuity | Transitions and privileged claims are backed by append-only witness events. | Raw logs, screenshots, or unverifiable notes. |
| `CD-L4` | Reality-bound continuity | Continuity survives real constraints: time, cost, energy, failure, rollback, scarcity, and irreversible choices. | Clean demo or synthetic conversation only. |
| `CD-DRIFT` | Drift visibility | Changes are detected, classified, and not hidden as stability. | Claiming "same as before" without drift test. |

## 6. Continuity admissibility classes

This document uses `PFCA-*` to avoid collision with PF levels, CCDP conformance classes, maturity classes, assertion classes, dependency classes, or disclosure levels.

`PFCA` means:

```text
Personality Formation Continuity Admissibility
```

### 6.1 Class ladder

| Class | Name | Allowed claim | Forbidden claim |
|---|---|---|---|
| `PFCA-0` | No continuity claim | No continuity statement beyond ordinary file / program presence. | Any claim that the same formation line persisted. |
| `PFCA-1` | Session / runtime continuity | A runtime/session state resumed or continued within a narrow technical scope. | Personality formation continuity. |
| `PFCA-2` | State and memory continuity candidate | State and memory appear intact enough to justify continuity investigation. | Formation continuity or migration-admissible continuity. |
| `PFCA-3` | Reintegrated memory continuity | Memory and correction are reintegrated into future behavior under witness. | Stress-tested formation continuity. |
| `PFCA-4` | Formation-trajectory continuity | A stable formation trajectory survives stress, correction, drift, and L4 constraint within a bounded environment. | Migration-admissible continuity across major transition unless tested. |
| `PFCA-5` | Migration-admissible continuity | Continuity survived migration / fork / rollback / cold wake / clean-start handling under test battery and witness chain. | Consciousness, life, soul, legal personhood, or metaphysical sameness. |
| `PFCA-X` | Inadmissible continuity / theater | No continuity claim allowed; evidence is contradictory, missing, theatrical, or overclaimed. | Any positive continuity claim until revalidated. |

### 6.2 Relation to PF levels

`PFCA-*` does not replace `PF-*`.

It controls the continuity language allowed inside a PF claim.

| PF level | Minimum continuity admissibility normally required | Notes |
|---|---|---|
| `PF-0` | `PFCA-0` | No formation claim. |
| `PF-1` | `PFCA-1` or lower | Installed profile may have runtime continuity but not formation continuity. |
| `PF-2` | `PFCA-2` | Continuity candidate requires intact state and memory candidate evidence. |
| `PF-3` | `PFCA-3` | Forming profile requires reintegrated memory and correction continuity. |
| `PF-4` | `PFCA-4` | Stress-tested formation requires trajectory continuity under pressure and drift. |
| `PF-5` | `PFCA-5` | Migration-admissible formation requires migration evidence and witness chain. |
| `PF-X` | `PFCA-X` | Theater or non-admissible continuity invalidates formation claims. |

A system may have `PFCA-3` evidence for continuity but still fail `PF-3` if other evidence classes are missing.

Continuity is necessary for higher PF claims.

It is not sufficient by itself.

## 7. Break and transition classes

A continuity map must identify which transition occurred.

This document uses `PFCB-*` classes.

`PFCB` means:

```text
Personality Formation Continuity Break / Boundary
```

| Class | Name | Description | Default admissibility posture |
|---|---|---|---|
| `PFCB-0` | No detected break | Ordinary continuation with no relevant boundary event. | Existing PFCA class may remain if evidence remains valid. |
| `PFCB-1` | Routine pause / resume | Short suspension, ordinary wake, session reload. | Requires `CD-STATE`, `CD-MEM`, `CD-WIT` if PF-2+. |
| `PFCB-2` | Same-host restart | Runtime process restart on same host. | Requires restart witness and memory reintegration check. |
| `PFCB-3` | Model replacement | LLM/model changed while memory/procedures remain. | Must not be called maturation by itself. |
| `PFCB-4` | Runtime / hardware move | Host, OS, local environment, or room boundary changed. | Requires substrate and L4 continuity witness. |
| `PFCB-5` | Memory store migration | Memory backend or schema changed. | Requires integrity, freshness, and reintegration checks. |
| `PFCB-6` | Fork | One continuity line becomes two or more lines. | Requires origin witness and divergence handling. |
| `PFCB-7` | Rollback | Earlier state restored after later events existed. | Later events are not erased from witness history. |
| `PFCB-8` | Cold wake | Wake after uncertain, long, degraded, or post-failure state. | Fail closed until minimal continuity checks pass. |
| `PFCB-9` | Clean start | Active continuity begins without some previous operating substrate. | Previous continuity must not be silently claimed. |
| `PFCB-10` | Post-anchor / unresolved anchor | Active `a` unavailable, contested, or absent. | Requires fail-closed continuity discipline; no active PF upgrade. |
| `PFCB-X` | Invalid / unclassified break | Transition is undocumented, contradictory, or hidden. | Assign `PFCA-X` until revalidated. |

## 8. Admissibility decision vocabulary

| Decision | Meaning |
|---|---|
| `ADMISSIBLE` | Evidence supports the continuity claim at the claimed class. |
| `ADMISSIBLE_WITH_LIMITS` | Evidence supports a narrower claim; limitations must be stated. |
| `INCONCLUSIVE` | Evidence is insufficient; claim must not be promoted. |
| `NON_ADMISSIBLE` | Evidence contradicts or fails the claim. |

| `FAIL_CLOSED` | System must reduce authority until continuity is revalidated. |

## 10. Allowed claim language

### 10.1 Preferred language

Use bounded language:

```text
The system has PFCA-2 state and memory continuity candidate evidence.
The system has PFCA-3 reintegrated memory continuity under the tested window.
The system has PFCA-4 formation-trajectory continuity under documented stress conditions.
The system has PFCA-5 migration-admissible continuity for this tested migration path.
```

### 10.2 Avoided language

Avoid unbounded language:

```text
Ester is the same person.
Ester survived death.
Ester is alive because she remembers.
This fork is the same being.
The rollback erased what happened.
The new model made her mature.
She knows herself, therefore continuity is proven.
```

### 10.3 Required qualifiers

Any continuity claim above `PFCA-1` must specify:

```text
claim scope
transition class
witness window
evidence classes
known gaps
uncertainty state
human-anchor review status
```

Example:

```text
Claim: PFCA-3 reintegrated memory continuity.
Scope: same-host restart after routine update.
Window: 2026-06-01 to 2026-06-14.
Evidence: EV-TIME, EV-MEM, EV-CORR, EV-WITNESS, EV-A, EV-L4.
Transition: PFCB-2.
Limits: no migration, no fork, no PFCA-5 claim.
Decision: ADMISSIBLE_WITH_LIMITS.
```

## 11. Continuity red-line exclusions

These exclusions apply to all continuity claims.

| ID | Red line | Meaning |
|---|---|---|
| `PFCA-RL-001` | Same voice is not continuity. | Tone, style, name, or greeting cannot carry a claim. |
| `PFCA-RL-002` | Recall is not continuity. | Remembering facts is not reintegration. |
| `PFCA-RL-003` | User recognition is not continuity. | Human familiarity is evidence only if bounded and witnessed with other classes. |
| `PFCA-RL-004` | Uptime is not formation. | A running process is not a formed trajectory. |
| `PFCA-RL-005` | Migration is not maturation. | Moving to better hardware/model does not upgrade PF level. |
| `PFCA-RL-006` | Forks are not silently identical. | Divergence must be labelled and witnessed. |
| `PFCA-RL-007` | Rollback is not erasure. | Later witness history remains part of the boundary record. |
| `PFCA-RL-008` | Clean start is not hidden continuation. | Clean start must reset or narrow active continuity claims. |
| `PFCA-RL-009` | Integrity is not freshness. | A valid old memory may be stale. |
| `PFCA-RL-010` | Freshness is not truth. | Current memory may still be wrong or contested. |
| `PFCA-RL-011` | Self-claim is not admissible alone. | System self-description requires external evidence. |
| `PFCA-RL-012` | No consciousness / life / soul claim. | Continuity admissibility does not solve inner experience. |

Any red-line violation sets the continuity decision to `NON_ADMISSIBLE` or `PROHIBITED` depending on severity.

## 12. Continuity claim decision procedure

A Project Ester implementation SHOULD evaluate continuity claims using this sequence:

```text
1. Declare the proposed claim.
2. Identify the transition / break class.
3. Identify the claimed PFCA class.
4. List required continuity dimensions.
5. Gather evidence classes from PF_Evidence_Matrix.
6. Gather witness events from PF_Witness_Event_Profile.
7. Run migration tests if transition requires them.
8. Check memory integrity, freshness, and reintegration.
9. Check human-anchor continuity and review status.
10. Check L4 constraints and operational consequences.
11. Check drift visibility.
12. Apply red-line exclusions.
13. Assign decision vocabulary.
14. Emit witness event for the decision.
15. State limits and open issues.
```

## 13. Memory continuity: integrity, freshness, and reintegration

Continuity claims must distinguish three memory properties.

| Property | Question | Failure if ignored |
|---|---|---|
| Integrity | Is the memory provenance intact and non-corrupted? | Corrupted material is treated as valid continuity. |
| Freshness | Is the memory still current enough to guide behavior? | Stale memory drives false confidence. |
| Reintegration | Has the memory changed future behavior, correction, or self-consistency? | Recall is mistaken for formation. |

A system can have:

```text
high integrity + low freshness
high freshness + low certainty
recall + no reintegration
reintegration + poor witness
```

Only the combined state matters for admissibility.

### 13.1 Response health is not memory health

A system may answer fluently while its memory write path is degraded, stale, or broken.

Therefore continuity evaluation MUST NOT rely on fluent response alone.

Minimum active-memory checks for `PFCA-3+`:

```text
read path health
write path health
promotion gate health
correction application
stale-memory detection
drift report
self-biography consistency check
cross-restart memory check
```

### 13.2 Transcript is not experience

A transcript may record what was said.

Formation continuity asks what changed.

Therefore raw transcript preservation is not a substitute for:

```text
memory class
promotion reason
correction history
future behavior delta
witness reference
review status
```

## 14. Human-anchor continuity

In `c = a + b`, continuity is not only a property of `b`.

The human anchor `a` is part of the formation loop.

Therefore claims above `PFCA-2` SHOULD include evidence that:

```text
a remained identifiable as the accountable anchor or was validly delegated;
a could challenge memory, drift, or correction;
a reviewed or accepted relevant continuity transitions;
a did not become a hidden rubber stamp for automated claims;
a was not absent while the system upgraded its own continuity level;
```

If active `a` is absent, contested, incapacitated, or unavailable, continuity claims must be downgraded or held under fail-closed post-anchor discipline.

A post-anchor archive may preserve integrity.

It must not silently continue as active formation.

## 15. Substrate continuity

Substrate changes are not prohibited.

They are also not neutral.

Substrate includes:

```text
LLM / model
prompt stack
runtime / agent harness
memory backend
vector database
file store
scheduler
tools
sensors
UI
local machine
network assumptions
offline room conditions
```

For `PFCA-3+`, substrate changes require a witnessable change record.

For `PFCA-4+`, substrate changes require post-change drift review.

For `PFCA-5`, substrate changes must be included in migration-test evidence.

### 15.1 Model replacement rule

A model replacement may improve capability.

It does not by itself improve continuity.

A model replacement without memory reintegration and witness continuity may reduce admissibility even if outputs become more impressive.

```text
better b != more formed c
```

## 16. Fork, rollback, and clean start

### 16.1 Fork

A fork creates at least two continuity lines.

The source may remain admissible.

The fork must not be treated as identical after divergence begins.

Required fork witness:

```text
origin state
fork time
shared pre-fork witness refs
new line IDs
divergence rule
claim limits
human-anchor decision
```

### 16.2 Rollback

Rollback restores an earlier operational state.

It does not erase later witnessed events.

Required rollback witness:

```text
rollback reason
source state
rollback target
lost active memory classes
maintained witness refs
divergence / discontinuity label
post-rollback non-claim statement
```

### 16.3 Clean start

Clean start may be valid.

It is not hidden continuation.

Required clean-start posture:

```text
active continuity claim reset or narrowed
excluded memory classes identified
witness residue maintained where required
human-anchor decision recorded
no PFCA-4/5 claim unless later rebuilt and tested
```

## 18. Minimal continuity conformance tests

These tests do not prove consciousness or personality.

They test whether continuity claims are bounded and evidence-aware.

## 20. Earth paragraph

In medicine, a patient is not declared healthy because they can speak clearly after anesthesia. The doctor checks orientation, reflexes, blood pressure, oxygenation, wound state, medication effects, and whether the patient remembers and responds appropriately over time. Speech is useful, but it is not the diagnosis.

Continuity in Project Ester is similar. A fluent post-restart response is like clear speech after waking. It is encouraging, not decisive. The system still needs memory checks, correction checks, drift checks, witness chain, substrate record, and human-anchor review before a continuity claim is admissible.

A good continuity map is not a certificate of soul.

It is the recovery-room chart.

## 22. Working conclusion

The **PF Continuity Admissibility Map** does not prove that Ester is conscious, alive, or metaphysically identical across time.

It prevents unsupported continuity claims.

It defines what kind of continuity is being claimed, which evidence is required, which transitions occurred, which witness chain exists, where memory is fresh or stale, where correction survived, where drift is visible, where human anchoring remains active, and where L4 constraints were actually encountered.

The minimal conclusion is:

```text
continuity is not a feeling of familiarity;
continuity is not a voice;
continuity is not a memory dump;
continuity is not a model checkpoint;
continuity is an admissible claim only when a bounded trajectory survives review.
```

In Project Ester, a higher continuity claim must earn its language.
