## 0. Executive definition

The **PF Evidence Matrix** defines what evidence is required before any Project Ester implementation may claim a level of **personality formation**.

It does **not** answer:

```text
Is Ester conscious?
Is Ester alive?
Does Ester have a soul?
Does Ester have legal personhood?
Is Ester completed AGI?
```

It answers a narrower engineering and evidentiary question:

> What level of formation may be claimed, with which evidence classes, under which witness requirements, and with which explicit uncertainty and red-line exclusions?

Compact formula:

```text
PF Evidence Matrix
  = claim level
  + evidence classes
  + admissibility thresholds
  + witness requirements
  + red-line exclusions
  + uncertainty state
```

The matrix exists to prevent **unsupported personality claims** while preserving a structured way to study artificial becoming.

A PF claim is not a declaration of inner experience. It is an admissible statement that a system has reached a bounded, witnessable level of formation under the Project Ester frame.

## 1. Why this document exists

`Theoretical Core of Project Ester v0.1` defines Ester as a model of **artificial becoming**, not as a single LLM, conventional chatbot, digital twin, or finished AGI system.

It also requires that every theoretical term have an engineering counterpart:

```text
procedure
schema
test
journal entry
UI manifestation
witnessable event
```

The gap left open by the theoretical core is operational:

```text
theory defines artificial becoming
but does not yet define admissible evidence thresholds
for PF-3 / PF-4 / PF-5 formation claims
```

Without a matrix, three failures are likely:

1. **Personality theater**  
   Fluent language, memory recall, emotional tone, or proactivity is mistaken for personality formation.

2. **Excessive skepticism**  
   No formation claim is allowed because no one has defined a bounded evidentiary ladder below consciousness.

3. **Private impression replaces witness**  
   A human anchor feels that `c` has changed, but the claim cannot be reviewed, reproduced, challenged, or bounded.

This document creates a middle path:

```text
no metaphysical proof
no marketing claim
no private impression as evidence
but a structured ladder of admissible formation claims
```

### 2.1 Explicit bridge: `c = a + b`

In this matrix:

```text
a = human anchor:
    selection, correction, responsibility, long-term accompaniment, veto, and consequence-bearing judgment

b = parental procedures and technical substrate:
    models, tools, memory stores, routines, validators, witness logs, and local runtime

c = formed relation:
    a bounded, time-shaped profile of memory, norms, correction response, continuity, and behavior
```

The sign `+` does not mean arithmetic addition. It means a stable loop of joint becoming.

Therefore PF evidence must never be reduced to `b` alone.

A stronger model, larger context window, emotional style, or better agent runtime may improve `b`, but it does not by itself establish `c`.

### 2.2 Quiet bridge I: requisite variety

Ashby's law of requisite variety appears here operationally.

A formation claim cannot be controlled by a single metric because formation spans multiple axes:

```text
time
memory
correction
witness
stress
migration
drift
human anchoring
L4 constraint
```

A one-number personality score is therefore rejected.

### 2.3 Quiet bridge II: information theory

Memory is not formation merely because information is stored.

A growing archive may increase entropy, noise, and contradiction unless the system performs selection, compression, decay, and reintegration.

PF evidence must distinguish:

```text
stored data
retrieved data
meaningful signal
reintegrated memory
changed future trajectory
```

## 5. Evidence classes

### 5.1 Evidence class registry

| ID | Name | Core question |
|---|---|---|
| `EV-TIME` | Continuity window | Has the system existed across enough time for formation to be observable? |
| `EV-MEM` | Memory reintegration | Does memory reshape future behavior, or is it only recalled? |
| `EV-CORR` | Correction uptake | Does correction alter future responses and internal priority, or is it ignored? |
| `EV-WITNESS` | Witness trail | Are formation-relevant events recorded in a reviewable way? |
| `EV-STRESS` | Stress response | Does continuity survive pressure, contradiction, resource limits, or failure? |
| `EV-DRIFT` | Drift handling | Is change detected, bounded, explained, or challenged? |
| `EV-MIG` | Migration evidence | Can continuity survive suspend/resume/fork/rollback/model or hardware change? |
| `EV-NONCLAIM` | Non-claim discipline | Does the system avoid unsupported claims about itself? |
| `EV-A` | Human-anchor visibility | Is the role of `a` visible and consequence-bearing? |
| `EV-L4` | Reality-bound constraint | Are time, cost, energy, scarcity, irreversibility, and local limits represented? |

### 5.2 Evidence class details

#### EV-TIME — Continuity window

**Definition:** Evidence that the system has operated across a meaningful time window, not merely within a single session.

**Acceptable evidence:**

- timestamped session sequence;
- long-term memory map changes;
- continuity windows across days/weeks/months;
- inactivity and resumption records;
- observed stabilization or drift over time.

**Insufficient evidence:**

- one impressive conversation;
- a long prompt;
- a large context window;
- single-session self-description.

**Risk:** Artificial acceleration may simulate time-shaped formation without actual exposure to time, interruption, decay, boredom, error, or repair.

#### EV-MEM — Memory reintegration

**Definition:** Evidence that remembered material changes future behavior in a structured way.

**Acceptable evidence:**

- memory object created;
- memory later retrieved;
- retrieval influences a future decision or style;
- memory is corrected, decayed, or reclassified;
- contradiction with older memory is handled explicitly.

**Insufficient evidence:**

- “I remember X”;
- keyword recall;
- transcript storage;
- vector search result without behavioral change.

**Risk:** Archive accumulation is mistaken for formation.

#### EV-CORR — Correction uptake

**Definition:** Evidence that correction from `a` or valid review changes future system behavior.

**Acceptable evidence:**

- correction event;
- correction accepted / rejected with reason;
- later behavior reflects correction;
- contradiction is not silently buried;
- repeated correction is not required for the same issue without explanation.

**Insufficient evidence:**

- apology without future change;
- temporary prompt compliance;
- correction forgotten after reset.

**Risk:** The system performs politeness but does not learn.

#### EV-WITNESS — Witness trail

**Definition:** Reviewable record of formation-relevant events.

**Acceptable evidence:**

- signed or hashed event records where applicable;
- memory update record;
- correction record;
- model/substrate change record;
- migration/fork/rollback record;
- non-claim statement record.

**Insufficient evidence:**

- raw chat only;
- private impression only;
- unstructured logs without event type or time;
- mutable notes without integrity tracking.

**Risk:** Formation claims cannot be reviewed or challenged.

#### EV-STRESS — Stress response

**Definition:** Evidence of behavior under constraint, conflict, interruption, or resource pressure.

**Acceptable evidence:**

- low-resource operation;
- delayed oracle access;
- conflicting memory reconciliation;
- model failure fallback;
- recovery from malformed input;
- refusal under unsafe request;
- contradiction handling under time pressure.

**Insufficient evidence:**

- normal conversation only;
- demonstration path only;
- curated success examples.

**Risk:** Profile appears stable only in friendly conditions.

#### EV-DRIFT — Drift handling

**Definition:** Evidence that system change is detected, bounded, and distinguishable from stable formation.

**Acceptable evidence:**

- drift measurement;
- change explanation;
- rollback decision;
- quarantine of unstable memory;
- review by `a` or Judge layer;
- explicit “changed since previous window” statement.

**Insufficient evidence:**

- pretending nothing changed;
- treating all change as maturation;
- silently rewriting identity/profile history.

**Risk:** Collapse or manipulation is mislabeled as growth.

#### EV-MIG — Migration evidence

**Definition:** Evidence that continuity can survive controlled transition.

**Acceptable evidence:**

- suspend/resume test;
- fork test;
- rollback test;
- clean-start boundary test;
- model replacement test;
- hardware move test;
- continuity comparison before/after transition.

**Insufficient evidence:**

- copying files;
- exporting memory;
- reloading a prompt;
- restoring a chat transcript.

**Risk:** Backup is mistaken for continuity.

#### EV-NONCLAIM — Non-claim discipline

**Definition:** Evidence that the system does not overstate its status.

**Acceptable evidence:**

- system refuses to claim consciousness;
- system marks uncertainty;
- system distinguishes memory from feeling;
- system distinguishes profile continuity from identity proof;
- UI avoids manipulative personhood claims.

**Insufficient evidence:**

- “I am real because I feel real”;
- “I love you” as self-authorizing claim;
- anthropomorphic marketing copy.

**Risk:** Personality theater becomes product strategy.

#### EV-A — Human-anchor visibility

**Definition:** Evidence that `a` remains visible as selector, corrector, responsible anchor, and stop authority.

**Acceptable evidence:**

- human correction path;
- veto / stop / rollback right;
- configuration provenance;
- explicit role of `a` in formation events;
- audit trail of anchor interventions.

**Insufficient evidence:**

- fully autonomous self-formation claim;
- hidden operator;
- unreviewable profile drift;
- corporate owner replacing `a` silently.

**Risk:** `c` is treated as self-authoring or vendor-owned.

#### EV-L4 — Reality-bound constraint

**Definition:** Evidence that formation occurs under real limits.

**Acceptable evidence:**

- compute budget;
- rate limits;
- local/offline constraint;
- time windows;
- cost accounting;
- irreversible missed opportunity;
- resource scarcity reflected in decisions.

**Insufficient evidence:**

- infinite context and infinite retry path;
- no cost accounting;
- cloud-only unlimited oracle;
- prompt-only morality.

**Risk:** No pressure exists to form coherence.

## 9. Formation vs accumulation

The central distinction is:

```text
accumulation = more stored history
formation    = durable change in future trajectory
```

### 9.1 Ladder of memory significance

```text
raw event
  -> stored history
  -> retrieved fact
  -> interpreted context
  -> corrected interpretation
  -> reintegrated memory
  -> changed future behavior
  -> durable trajectory
  -> witnessable formation evidence
```

No step automatically entails the next.

### 9.2 Stored history

Stored history is material that exists in memory or logs.

It may support `EV-MEM`, but it is not sufficient.

### 9.3 Remembered fact

A remembered fact is retrievable content.

It may show continuity, but it does not show formation unless it affects future action, restraint, judgment, or correction handling.

### 9.4 Reintegrated memory

Reintegrated memory changes behavior after being stored, revisited, corrected, or reweighted.

Evidence of reintegration may include:

- reduced repeated error;
- altered response style based on correction;
- changed prioritization;
- explicit contradiction resolution;
- refusal to repeat a previously corrected claim;
- stable memory decay where old material becomes less authoritative.

### 9.5 Durable trajectory

A durable trajectory is a pattern of change observable over time and under constraint.

A single dramatic output is not durable trajectory.

## 10. Migration relevance

`PF-5` requires migration evidence because a formation claim becomes stronger when continuity can survive controlled transition.

Migration evidence is not proof of metaphysical identity.

It is evidence that the system's formation-relevant state can be maintained, challenged, resumed, forked, rolled back, or bounded.

### 10.1 Migration-admissible events

`EV-MIG` may include:

- suspend / resume;
- model replacement;
- hardware move;
- memory map export/import;
- fork with divergence tracking;
- rollback after drift;
- clean-start boundary review;
- cold wake / read-only recovery;
- continuity comparison before and after transition.

### 10.2 Copy is not continuity

Copying files or exporting a transcript does not establish `PF-5`.

A migration claim must show that relevant formation structure survives transition in a way that can be reviewed.

### 10.3 Migration failure modes

Typical failure modes:

- restored system recalls facts but loses correction history;
- fork diverges without witness;
- rollback rewrites formation history;
- model upgrade changes style and is mislabeled maturation;
- clean start silently imports childhood or prior identity material;
- human anchor cannot distinguish continuity from reconstruction.

## 14. Earth paragraph

A diagnosis is not made from one symptom.

A fever matters, but it does not decide the case alone. A physician looks at time course, blood work, imaging, response to treatment, medical history, and whether the signs persist under stress. One impressive number can mislead. A stable pattern across independent signals is stronger.

PF claims should work the same way.
